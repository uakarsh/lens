Lens — Error Analysis for Image Classifiers

What's Actually Wrong With Your Model

Akarsh Upadhyay  ·  May 2026

The Problem

Standard model debugging doesn't tell you much

  • You train, you check accuracy, you look at a confusion matrix
  • Model underperforms → try more epochs, different LR, different architecture
  • You're guessing at the root cause rather than reading it from the data
  • The core issue: not all failures are the same kind of problem — and they need completely different fixes
The Problem

Three kinds of failures — three different fixes

  • Data quality — mislabeled or ambiguous image, a human would also get it wrong → fix the data
  • Visual gap — the model hasn't seen this visual pattern during training → fix with augmentation
  • Hard task — genuinely ambiguous for this architecture → accept it or change the model
Treating all three the same wastes time on the wrong fix. If 56% of your failures are data quality issues, more training doesn't help.
The Approach

Lens — structured per-failure analysis

01
Train and collect failures
Fine-tune model, save every misclassification with top-3 predictions and confidence
02
Generate hypothesis per failure
Send each failure image to a local VLM. Ask: data issue? visual cause? suggested fix?
03
Cluster each dimension independently
Embed and cluster data issues, visual causes, and fixes as separate axes — not one blob
04
Human review
Annotator pages through failures, records agree/disagree, sees cluster summaries on a dashboard
Setup

CIFAR-10 ResNet-18 baseline

  • Model: ResNet-18 (ImageNet pretrained), FC replaced for 10 classes, 3 epochs, no augmentation
  • VLM: qwen3-vl:8b via Ollama — runs locally, no external API calls
  • Sample: 150 failures sampled for analysis (15 per class)
  • Embeddings: BAAI/bge-base-en-v1.5 + KMeans, k=15–30 with silhouette selection
  • Stack: PyTorch · Ollama · SentenceTransformers · scikit-learn · FastAPI

Findings

Val loss rises for cat/dog — but it's not overfitting

  • Cat and dog val loss rises after epoch 2 — looks like a training problem
  • Tried: dropout on FC layer → accuracy dropped
  • Tried: two-phase training (freeze backbone, then unfreeze) → accuracy dropped further
  • Tried: lower backbone LR → same result
  • Conclusion: the val loss rise is boundary ambiguity in the dataset — the model is doing about as well as it can on these images
This is exactly the kind of insight you want before spending a week on training experiments. The problem is the data, not the training.
Findings

45% of failures flagged as ambiguous images

Data issue tags (68 / 150)
  • 68 ambiguous_image — cropping removes key features, background texture bleed, images a human would also get wrong
  • 66 model_error — image looks fine, model just got it wrong
What this means
  • Ambiguous images signal the same kinds of conditions the visual cause clusters surface
  • They're a guide to what augmentations might help
  • VLM also produced 16 annotation_issue flags — manual review suggests most are wrong

The Hard Part

Hypothesis generation is the bottleneck

  • Correct — green car predicted as frog at 94% conf: "saturated green background dominates the frame." Verifiable, actionable fix follows directly
  • Almost right — bird/airplane confusions: VLM correctly identifies the right failure class, but gets the specific visual reason wrong
  • Slightly wrong — aircraft carrier as airplane: structural reasoning holds up, but invents background details
  • Fully hallucinated — ship as truck: describes "uniform blue background," builds entire causal chain on it. Background is dark grey gravel.
The VLM produces fluent, structured, confident output — which makes hallucinations harder to catch, not easier. Prompt engineering and larger models are the next lever.
What's Next

Planned improvements

  • Better hypothesis prompts — describe what you see before reasoning; ask for explicit uncertainty; chain-of-thought grounding
  • Larger VLM — qwen3-vl:72b or a cloud model; the 8b model hallucinates at a rate that limits downstream trust
  • Wire human verdicts back in — filter noisy hypotheses before clustering; weight clusters by reviewer confidence
  • Iteration 2 training — run with head occlusion, background texture, crop augmentations; compare per-class F1 vs baseline
  • Cross-run comparison — side-by-side cluster diff between baseline and iteration 2
Summary

Key takeaways

  • Structured per-failure hypotheses (data / visual / fix) + clustering is a sound architecture for error analysis
  • Hypothesis generation is the hard part — the VLM hallucinates image details at a non-trivial rate, which limits how much you can trust individual hypotheses
  • Cluster-level signal can still be directionally useful even when per-image explanations are unreliable
  • Human review is not optional — it's the only way to know how much to trust the output
